# 第 15 章 长程任务的工程难题

> **问题陈述**：第 13–14 章构建了循环工程的拓扑和控制结构。然而，这些设计在面对**长程任务**（需要数十到数百步、持续数小时甚至数天的 Agent 任务）时会暴露出新的工程难题——上下文会溢出窗口，Token 预算会耗尽，模型更新会导致回滚失败，Agent 会陷入无限循环。本章处理这些"长程"带来的工程难题：上下文滚动与状态持久化、资源预算、以及失败模式分类与恢复。

---

## 15.1 上下文滚动与状态持久化

长程任务的核心矛盾是：模型能"记住"的内容（上下文窗口）有限，但长程任务需要 Agent 在远超上下文窗口的时间内保持状态。

### 15.1.1 工作区作为持久状态

第 10 章的工作区（workspace）功能在第 9 章最小 Harness 中是一个文件系统目录——Agent 可以在其中任意读写文件。在长程任务中，工作区从"可选功能"升级为**核心状态存储**。

**文件系统的 CRDT 化思路。** 当 Agent 的多个分支（子 Agent、并行步骤）同时修改工作区时，传统文件系统的"最后写入覆盖"策略可能丢失数据。CRDT（Conflict-free Replicated Data Type）是一种允许多方同时编辑、自动合并冲突的数据结构。将工作区 CRDT 化意味着：每个文件操作都附带一个逻辑时钟（Lamport 时钟或 Vector Clock），Harness 在聚合阶段自动合并冲突。CRDT 化的代价是实现复杂度高——不是所有文件类型都适合自动合并（如二进制 PDF 文件）。

对于大多数工程团队，CRDT 化的投资回报率不如**操作序列化**（将所有文件操作串行化，避免并发冲突）。操作序列化的实现更简单：在并行子 Agent 模式中，共享文件的所有写入操作由父 Agent 在聚合阶段统一执行，子 Agent 只读不写。这才是实际的工程建议。

> **工程原则 1（长程任务状态持久化原则）**：Agent 的长程状态应按"不依赖 LLM 上下文窗口"的方式持久化。工作区文件系统是首选（不依赖模型），结构化记忆（第 7 章）是补充。任何"只在上下文中保存"的状态都会在 Agent 会话结束时丢失。

### 15.1.2 任务恢复

任务恢复是从中断或失败中继续执行的能力——对长程任务至关重要。

**恢复时的上下文重建。** 任务恢复的核心挑战是上下文重建：中断发生时，$P/H/R/O/S$ 五元组中的 $H$（对话历史）可能已经包含了数百步的交互，全部恢复将超出上下文窗口。解决方案：在恢复时对 $H$ 做摘要（使用第 7 章的增量摘要方法），将数百步的 $H$ 压缩为一页摘要。$R$ 和 $O$ 直接从工作区的最新文件状态中读取（不需要恢复每一步的工具调用记录）。

**模型版本变更下的兼容性。** 生产环境中模型版本更新频繁（每月数次）。Agent 在第 10 步中断，模型在第 11 步前更新——恢复时第 1-10 步的状态快照是由旧版本生成的，第 11 步的新调用使用新版本。风险：新版本的输出可能不一致（即使 seed 相同）。缓解方法：①**固定模型版本**——长程任务的整个生命周期内锁定模型版本（推荐）；②**版本感知快照**——在状态快照中记录模型版本，恢复时检查版本是否一致，不一致时触发"预热"步骤（用新版本重新执行上一步，验证输出是否可接受）。

## 15.2 资源预算

长程任务比短任务更需要严格的资源预算——短任务浪费 Token 的代价很小，长任务浪费 Token 可能让任务因预算耗尽而失败。

### 15.2.1 Token 预算分配

Token 预算应在**计划阶段**而非执行阶段分配。在目标分解时（第 14 章），为每个子目标估算 Token 消耗，汇总得到总预算。如果总预算超出用户设定的上限，在计划阶段就压缩或合并子目标，而非在执行到一半时因预算耗尽而终止。

预算估算方法：根据历史数据，每个 ReAct 循环步平均消耗 2,000–5,000 Token（输入）+ 500–2,000 Token（输出）。

**定义 15.1（Token 预算）**：一个 $N$ 步 Agent 任务的 Token 预算 $B$ 定义为：
$$B = N \times (B_{\text{input}} + B_{\text{output}}) \times \alpha$$
其中 $N$ 为估计执行步数，$B_{\text{input}}$ 为单步平均输入 Token 数（从历史数据获取），$B_{\text{output}}$ 为单步平均输出 Token 数，$\alpha$ 为安全系数（推荐 1.3–1.5）。预算应在计划阶段而非执行阶段分配。

### 15.2.2 时间预算与并发

时间预算决定任务需要在多少时间内完成。如果用户期望 Agent 在 5 分钟内完成一个预计需要 20 分钟的任务，Harness 需要决定是否启用并发（子 Agent 并行执行独立子目标）。并发的加速效果不是线性的——Amdahl's Law 限制了可并行的上限：$Speedup = 1 / ((1 - p) + p / n)$，其中 $p$ 为可并行部分的比例，$n$ 为并行数量。如果任务中 60% 可并行，即使使用 10 个子 Agent，最大加速比也只有 2.17 倍。

### 15.2.3 金额预算与降级策略

金额预算直接关联 Token 消耗。当实际 Token 消耗超过预算时，Harness 应启动降级策略，而非直接终止。降级策略按严重程度递增：①**模型降级**——将高成本模型（如 GPT-4o）降级为低成本模型（如 GPT-4o-mini），准确率可能下降 5–10% 但 Token 成本降低 80%；②**上下文压缩**——将 $H$ 的滚动窗口缩小 50%（从保留 16 轮对话降为 8 轮），减少输入 Token；③**终止并返回部分结果**——保存已完成的子目标结果，向用户返回"任务已部分完成"。

> **反方观点**：降级策略中的"模型降级"可能引入不一致——Agent 前半段用 GPT-4o 做了高质量分析，后半段用 GPT-4o-mini 降低了质量，最终输出可能出现前后质量不匹配。工程建议：降级应在**子目标边界**处执行（而非在循环中间切换），确保至少每个子目标的内部质量是一致的。

## 15.3 失败模式分类与恢复

长程任务比短任务面临更多类型的失败。每种失败需要不同的检测和恢复策略。

### 15.3.1 死循环检测

Agent 在循环中不断重复相同的操作模式，无法推进任务。死循环是长程任务中最常见、最消耗 Token 的失败模式。

**动作哈希指纹。** 为每个工具调用生成一个指纹 $H(\text{tool\_name}, \text{args}_{\text{normalized}})$，其中 $H$ 是一个哈希函数（如 SHA-256），args_normalized 是将参数排序序列化后的标准化字符串。Harness 维护一个滑动窗口（最近 $W$ 步）的指纹历史。如果当前步的指纹与窗口内的某个指纹匹配，触发死循环告警。

**N-gram 重复阈值。** 动作哈希指纹只能检测**完全相同的**重复调用。实际中的死循环可能表现为"模式重复"——Agent 在反复搜索、读取、思考、再搜索，每次的参数不同但工作流程相同。N-gram 检测方法：将最近 $L$ 步的 Action 类型（非参数）构建为 N-gram 序列，如果某个长度为 $N$ 的序列在滑动窗口中出现超过 $T$ 次，判定为死循环。推荐配置：$N=3, L=15, T=3$（即最近 15 步中，如果同一个 3 步模式出现 3 次以上，判定死循环）。

### 15.3.2 幻觉级联

长程任务中，模型一次轻微的幻觉（如错误地认为"函数 X 已被删除"）可能导致后续步骤基于这个错误的假设构建推理——这就是幻觉的级联效应。

**早期止损信号。** 检测幻觉级联的早期信号是**跨步事实一致性检查**：将第 $t$ 步的 LLM 输出中陈述的事实与第 $t-1$ 步和第 $t-2$ 步的 Observation 做对比。如果第 $t$ 步陈述了一个与之前 Observation 矛盾的事实，触发幻觉告警。例如，第 5 步的 Observation 包含"函数 foo() 存在于 main.py 第 10 行"，但第 7 步的 Thought 说"由于 foo() 不存在，我决定创建一个新的"——这是一个明确的幻觉信号。

> **真实失败案例**：某 Agent 在第 4 步读取了一个配置文件的旧版本（缓存未过期），断言"数据库连接配置为 localhost"。后续 6 步基于这个错误的配置构建了完整的数据库迁移计划。第 10 步执行连接测试时才发现配置已被改为远程连接。如果 Harness 在第 5 步执行了跨步事实一致性检查（对比第 4 步的 Observation 与用户最新配置文件），这个幻觉级联可以在第 5 步就被终止。

### 15.3.3 工具反复失败

Agent 重复调用同一个工具，每次都因相同的原因失败。

**退避与替代工具切换。** Harness 应记录每个工具的连续失败次数。当失败次数超过 $F_{\max}$ 时（推荐 $F_{\max}=3$），执行退避策略：让 Agent 先执行其他步骤，等待至少 60 秒后重试。如果重试仍然失败，Harness 应建议 Agent 切换到替代工具（如搜索工具超时时，切换到更简单的 grep 实现；LLM 调用超时，切换到次选模型）。

### 15.3.4 目标漂移

第 14 章的目标漂移检测适用于短到中等长度的任务。对于长程任务，漂移检测周期需要调整——每 $K$ 步检查一次可能遗漏漂移恢复后再次漂移的反复模式。

**周期性目标回锚。** 对于预计超过 30 步的长程任务，推荐使用"周期性目标回锚"而非全局的目标一致性检查。回锚策略：每完成一个子目标后，将当前输出与原始目标做比对，并生成一个"回到正轨"的确认。回锚的优点是每个子目标的结束都是天然的检查点——不需要额外的 LLM 调用来做一致性检查。

## 附：长程任务评估指标表

| 指标名称 | 定义 | 度量方法 |
|---------|------|---------|
| 任务完成率 | 长程任务在预算耗尽前完成的比例 | 成功完成的任务数 / 总启动任务数 |
| 恢复成功率 | 中断后任务恢复并成功完成的概率 | 恢复后成功完成的次数 / 总恢复尝试次数 |
| 死循环检测率 | 死循环被检测并在 $M$ 步内纠正的比例 | 被检测的死循环数 / 实际发生的死循环数 |
| 模型降级影响率 | 模型降级导致的输出质量下降幅度 | 降级前后的准确率差异（%） |
| Token 预算偏差率 | 实际 Token 消耗与预算的偏差 | （实际 - 预算）/ 预算，越低越好 |

## 开放问题

1. **工作区 CRDT 化的实际价值。** 对于大多数 Agent 任务（代码修改、文件读写），CRDT 化的复杂度是否值得？哪些类型的任务确实需要 CRDT（多 Agent 协作编辑同一个文档）？

2. **模型版本锁定 vs 持续更新。** 长程任务锁定模型版本可以获得可重现性，但也意味着无法享受新版本的 Bug 修复和性能提升。是否存在一种"平滑更新"策略——在子目标边界切换模型版本，容忍 1-2 步的输出波动？

3. **死循环检测的误报率。** 压缩图片或编译大型项目等任务天然需要重复调用相同工具（如连续压缩 100 张图片）。死循环检测算法如何区分"合法的重复"和"死循环"？

4. **Token 预算的动态调整。** 固定预算过于刚性——如果任务比预期困难，Agent 应在什么条件下允许超预算？需要一个人工确认的"预算追加"入口吗？

## 练习

### 思考题

1. 你的 Agent 需要执行一个 50 步的长程任务：为整个代码库添加单元测试。设计它的 Token 预算分配方案：每个阶段的预算占比是多少？安全系数 $\alpha$ 取多少？哪些步骤可以降级？

2. 如果工作区 CRDT 化的复杂度你无法接受，但你又需要多个子 Agent 同时编辑同一个文件——你如何设计一个"串行化写入"的替代方案？写入顺序由谁决定？冲突如何检测？

3. 死循环检测算法在连续压缩 100 张图片的任务中触发了报警——你的算法如何区分"合法重复压缩"和"死循环压缩"? 设计一个区分方案（提示：对比每次工具调用的输入是否相同）。

### 动手题

1. 实现动作哈希指纹检测：给定一系列工具调用记录（工具名+参数），使用 SHA-256 生成指纹，检测最近 10 步中是否出现重复指纹。验收标准：当同一个工具和参数在第 5 步和第 8 步重复出现时，输出"死循环告警"。

2. 实现一个简单的 Token 预算监控器：在 Agent 每步循环后累加 Token 消耗，触发降级链（模型降级→上下文压缩→终止）。验收标准：在一个 10 步的任务中，设置预算为只能完成 6 步，验证第 6 步后触发模型降级，第 8 步后触发终止。

3. 实现周期性目标回锚：每完成一个子目标后，将当前输出与原始目标做比对。验收标准：给定 3 个子目标和原始目标，输出每轮回锚的对比结果（"一致"或"偏离"）。
