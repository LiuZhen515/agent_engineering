# 附录 B：常见反模式速查卡

> 以下反模式按四大工程分类组织。每条记录包含：**症状**（识别特征）、**后果**（工程损失）、**解决方案**（推荐做法）。
> 正文中对应章节以 [ChX] 标注。

---

## 一、提示词工程（Prompt Engineering）

### B.1 角色锚定幻觉

- **症状**：给模型分配的角色与实际任务无关（如让模型扮演"莎士比亚"写代码审查报告），或在不需要专业知识的简单任务上也使用角色锚定。
- **后果**：角色锚定增加了 Token 消耗，但没有提升输出质量。对于常见任务（翻译、摘要），角色锚定几乎没有增益（[Ch3] 反方观点）。
- **解决方案**：仅在需要领域知识的专业任务（法律文书、医疗诊断）中使用角色锚定。简单任务跳过 Role 步骤，直接给 Task + Format。

### B.2 Token 对齐幻想

- **症状**：假设模型的分词器与人类对"词语"的直觉一致，在提示词中使用"自然语言行"而非"Token 对齐行"来对齐格式。
- **后果**：格式对齐失效——模型的分词器可能在任意位置截断 Token，导致视觉对齐和实际对齐不一致（[Ch2] §2.1.1）。
- **解决方案**：使用视觉分隔符（如 `---`、`===`、空行）而非空格对齐。需要精确对齐时，先用目标模型的分词器验证。

### B.3 过度依赖 logprobs

- **症状**：假设所有模型都支持 `logprobs` 参数，并将其作为通用提示词质量诊断工具。
- **后果**：脚本在 DeepSeek 等不支持 logprobs 的模型上崩溃（[Ch2] Listing 2.1）。
- **解决方案**：使用前先探测模型是否支持 logprobs。如果不支持，切换模型的 API 版本或改用其他诊断方法（如输出一致性检查）。

### B.4 隐式 Chain-of-Thought

- **症状**：在提示词中写"让我们一步一步思考"但不提供具体推理路径模板，依赖模型自行决定推理步骤。
- **后果**：模型可能产生幻觉或逻辑跳跃，尤其在需要领域特定公式或业务规则的场景中（[Ch3] 反方观点）。
- **解决方案**：在 CoT 提示词中明确指定推理路径（"第一步：提取事实；第二步：应用规则 X；第三步：得出结论"）。对于通用推理任务，依靠模型内置推理能力即可。

### B.5 一次性提示词

- **症状**：提示词没有版本管理，不做回归测试，每次修改直接在生产环境生效（[Ch4] §4.1）。
- **后果**：修改后质量下降无法追溯，回滚困难，团队协作混乱。
- **解决方案**：使用 Golden Set 做回归测试（Listing 4.2），提示词版本管理（定义 4.1），修改后准确率不低于阈值才部署。

## 二、上下文工程（Context Engineering）

### B.6 上下文填满错觉

- **症状**：将模型声称的最大上下文长度当作"有效上下文长度"，无限制地向上下文中填充内容。
- **后果**：中间的 Token 被注意力稀释（[Ch2] Lost in the Middle），新增信息不仅不提升质量，反而稀释已有信息。超过 32K Token 后边际收益趋近于零（[Ch5] §5.1.2）。
- **解决方案**：设置有效使用上限（建议最大长度的 50-70%），关键信息放在开头或末尾。

### B.7 单一检索依赖

- **症状**：RAG 系统只使用一种检索方法（如仅向量检索或仅 BM25），没有多路融合。
- **后果**：单一方法的召回率上限比多路融合低 10-20%（[Ch6] 工程原则 1）。
- **解决方案**：至少使用两种互补方法（向量 + BM25），通过 RRF 融合结果（Listing 6.1）。

### B.8 语义切分过度追求

- **症状**：花费大量精力设计"语义完整"的文档切分策略，认为固定窗口切分会破坏语义连续性。
- **后果**：语义切分器增加了系统复杂度，但在向量检索时代收益有限（[Ch6] 反方观点）。相邻块的语义重叠已足以弥补切分截断。
- **解决方案**：采用固定窗口 + 充分重叠的切分策略（重叠率 10-20%），在多数场景下优于复杂的语义切分器。

### B.9 记忆填满症

- **症状**：Agent 记住所有历史对话，从不压缩或遗忘，长期记忆最终被噪声填满。
- **后果**：上下文窗口被无关信息占据，Token 成本持续增长，关键信息被稀释（[Ch7] §7.3.1）。
- **解决方案**：实施时间衰减（指数衰减曲线），定期做增量摘要（定义 7.1），使用访问强化机制。

### B.10 反思过度

- **症状**：对所有用户都启用主动反思，且反思频率过高（每次对话后都反思）。
- **后果**：Token 成本大幅增加（每次反思消耗一次 LLM 调用），但只有约 30% 的用户能从中获得可感知的体验提升（[Ch7] 反方观点）。
- **解决方案**：仅对高频用户启用反思，低频用户依赖被动事件流。

### B.11 工具调用历史吞噬上下文

- **症状**：将每次工具调用的完整 JSON 请求体和返回结果全部写入上下文，不裁剪、不摘要。
- **后果**：工具调用结果 $O$ 可能占数万 Token，污染上下文并稀释注意力（[Ch8] §8.2.1）。
- **解决方案**：对工具结果做字段白名单裁剪、长度截断和去重（[Ch8] §8.2.1）。仅保留最终结果，丢弃中间过程详情。

### B.12 五元组合并

- **症状**：将 $ R $（检索结果）和 $ O $（工具输出）合并进 $ H $（对话历史），认为"最终都写入了对话历史，不需要单独管理"。
- **后果**：丢失可归因的工程调试信息——无法判断准确率下降是检索质量差还是模型推理逻辑的问题（[Ch5] 反方观点）。
- **解决方案**：保持五元组分离，为每个分量设置独立的生命周期管理策略。

## 三、驾驭工程（Harness Engineering）

### B.13 Harness-Framework 耦合

- **症状**：将 Harness（运行时环境）的功能内建于 Framework（开发框架）中，不做分离。
- **后果**：原型阶段效率高，但生产环境面临扩展性瓶颈——权限控制、多租户隔离、企业级可观测性无法独立演进（[Ch9] 反方观点）。
- **解决方案**：原型阶段可容忍耦合，生产阶段将 Harness 分离为独立层（[Ch9] 定义 9.1）。

### B.14 工具参数缩写病

- **症状**：工具参数使用缩写名（如 `q`、`n`、`lang`），参数描述过短（< 10 Token）或缺失。
- **后果**：模型无法理解参数含义，调用准确率显著下降（[Ch8] §8.1.1）。
- **解决方案**：参数名使用 `snake_case` 全拼（2-4 个英文单词），描述中加入典型值示例（"例如：'transformer attention mechanism'"）。

### B.15 全黑名单安全策略

- **症状**：Agent 的权限控制使用黑名单模式——默认允许所有操作，仅禁止明确列入黑名单的操作。
- **后果**：新出现的安全威胁不在黑名单中，Agent 可能执行危险操作（[Ch10] §10.2）。
- **解决方案**：生产环境使用白名单模式——默认拒绝所有操作，仅允许明确授权的工具和操作。

### B.16 审批疲劳

- **症状**：Agent 的每个工具调用都要求用户确认，包括确定性的读取操作。
- **后果**：用户频繁点击确认，产生"确认疲劳"，最终可能不经审查就盲目批准（[Ch11] §11.2.2）。
- **解决方案**：只对高危操作（写文件、执行 Shell、删除操作）触发审批。确定性读取操作自动执行。对已批准的操作类型做信任缓存。

### B.17 可观测性盲区

- **症状**：Harness 自身没有可观测性——无法追踪 Agent 的决策链、Token 消耗和成本归因。
- **后果**：调试困难——不知道 Agent 做错了什么、哪一步消耗了最多 Token（[Ch12] §12.1）。
- **解决方案**：实现 Span 最小集（定义 12.1），记录 Token 账本（§12.1.3），为每次决策保留状态快照。

### B.18 工具描述自反馈过拟合

- **症状**：基于少量失败案例自动修改工具描述，导致描述适配了特例而劣化了通用情况（[Ch16] 反方观点）。
- **后果**：工具描述在几次修改后变得冗长、矛盾，模型反而更难正确调用。
- **解决方案**：自动修改需在 Golden Set 上经过回归测试（`test_no_regression_after_change`），通过率不低于阈值时才部署。

## 四、循环工程（Loop Engineering）

### B.19 死循环无检测

- **症状**：Agent 循环中没有死循环检测机制，Agent 反复调用同一个工具得到相同结果，直到 Token 耗尽。
- **后果**：大量 Token 浪费在无效循环中，任务永远无法完成（[Ch15] §15.3.1）。
- **解决方案**：实施动作哈希指纹（SHA-256 归一化参数）和 N-gram 重复阈值检测（N=3, L=15, T=3）。

### B.20 幻觉级联无止损

- **症状**：Agent 一次轻微幻觉导致后续所有步骤基于错误假设构建推理，且系统没有检测机制。
- **后果**：越往后错误越大，最终输出完全偏离事实（[Ch15] §15.3.2）。
- **解决方案**：实施跨步事实一致性检查——将第 $ t$ 步的输出与第 $ t-1$ 步和 $ t-2$ 步的 Observation 做对比，矛盾时触发幻觉告警。

### B.21 Plan-Execute 过度使用

- **症状**：对所有任务都使用"先计划再执行"的 Plan-Execute 模式，包括只需要一步的简单任务。
- **后果**：简单任务中增加了不必要的 Token 开销和延迟（[Ch13] 反方观点）。
- **解决方案**：给 Agent 一个任务复杂度评估器——简单任务走 ReAct，复杂任务走 Plan-Execute。

### B.22 隐式约束自作主张

- **症状**：Agent 在目标分解阶段自行推断大量隐式约束，并将它们作为"硬约束"而非"建议"。
- **后果**：Agent 可能添加用户不想要的约束（如用户只说"审查 PR"，Agent 自行添加"不要删除注释"的约束），导致执行结果与用户意图偏离（[Ch14] 反方观点）。
- **解决方案**：隐式约束以"建议"形式呈现，最终由用户确认或拒绝。

### B.23 降级边界混乱

- **症状**：Token 预算耗尽时，在 Agent 循环的中间位置切换模型（如从 GPT-4o 降级到 GPT-4o-mini）。
- **后果**：同一任务前后输出质量不匹配——前半段高质量分析 + 后半段低质量输出（[Ch15] 反方观点）。
- **解决方案**：降级在子目标边界处执行，确保每个子目标的内部质量一致性。

### B.24 评估格式猜测

- **症状**：LLM-as-Judge 的评分解析逻辑假设评分总是 0-10 格式，但 Judge 模型可能返回 0-1、0-5、"8/10"等不同格式。
- **后果**：评分解析错误，导致评估结果偏差（[Ch4] Listing 4.3 注释）。
- **解决方案**：在 Judge 提示词中明确指定输出格式（"请给出 0-1 之间的分数，仅输出数字"），使用更鲁棒的正则表达式。

## 五、跨工程反模式

### B.25 基准污染麻木

- **症状**：因为公开基准存在污染（数据泄露到训练集），完全放弃使用基准评估。
- **后果**：失去了跟踪相对改进的重要工具（[Ch18] 反方观点）。
- **解决方案**：继续使用基准评估**相对改进**——如果优化后提升 5%，这个提升在干净数据上很可能同向存在。绝对数值不值得信任，但相对趋势有意义。

### B.26 内建运行时

- **症状**：从头自研 DSL 来管理提示词模板，投入大量精力维护 DSL 编译器。
- **后果**：DSL 的投资回报率在大多数团队中为负（[Ch4] 反方观点）。维护 DSL 的时间本可以花在优化提示词本身。
- **解决方案**：回归通用模板引擎（Jinja2）+ 良好的文件组织（YAML 拆分组件）。结构化提示词文件的组织方式已足够，引入新语言门槛较高。

### B.27 Model Agnostic 陷阱

- **症状**：假设提示词在所有模型上表现一致，不针对目标模型做适配。
- **后果**：同一提示词在 GPT-4o 和 GPT-4o-mini 上的表现可能差异显著（[Ch4] §4.2）。
- **解决方案**：定期对 Golden Set 重新运行评估，跟踪准确率变化趋势。当准确率下降超过预设阈值（如 5%）时，触发告警并启动提示词适配流程。
