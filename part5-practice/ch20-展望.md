# 第 20 章 展望

> **问题陈述**：前 19 章的四层工程框架是建立在当前（2026 年）模型能力基础上的。然而，模型能力正在高速进化——上下文窗口每年翻 10 倍，推理模型改变了 CoT 的编写方式，原生工具调用正在简化 Harness 的设计。这些进化会如何重塑四层工程的边界？哪些工程层会消失，哪些会融合，哪些会获得新的重要性？本章尝试回答这些面向未来的问题。

---

## 20.1 模型能力提升如何重塑四种工程的边界

模型能力的每一次跃升，都会"吞噬"一部分当前由工程方法解决的问题。识别哪些工程问题正在被模型吞噬，哪些依然存在，是有效分配工程投入的前提。

### 20.1.1 长上下文是否会让 RAG 消失

当前趋势：模型的上下文窗口从 2023 年的 4K Token 增长到 2026 年的 2M Token（Gemini 1.5 Pro）。如果这个趋势持续，理论上整个知识库可以一次性"喂入"上下文，RAG 的索引和检索层似乎变得多余。

但实际情况比"消失"更复杂：

**RAG 仍然必要的场景**：①**成本优化**——2M Token 的预填充成本约 $3-5（GPT-4o 定价），而 RAG 检索的成本约 $ 0.001。对于高频查询，RAG 的成本优势是 3,000:1；②**权限控制**——长上下文无法区分哪些内容给定用户的可见权限。RAG 的权限过滤（第 17.1.2 节）在长上下文场景中仍然必要；③**时效性**——知识库每小时更新，但上下文是快照。RAG 可以实时感知更新，长上下文则永远落后于最新状态。

**RAG 在长上下文时代的演化方向**：RAG 不会消失，但会从"补足上下文窗口的不足"转向"管理上下文窗口中的信息密度"。RAG 的角色从"能否被检索到"变为"检索到了如何排布以最大化注意力利用率"——这与第 5 章的四维质量指标直接呼应。

> **反方观点**：如果模型提供商提供"按需加载"的原生长上下文（如只在推理时从远程存储读取部分内容，而非一次性加载全部 Token），RAG 的检索层可能会被模型内置的检索机制替代。届时，RAG 的生存空间只剩下权限控制和成本优化两个方向。

### 20.1.2 推理模型如何吞噬 CoT 提示词

o1、o3、DeepSeek-R1 等推理模型在训练阶段内置了 CoT 推理能力。用户在 Prompt 中写"让我们一步一步思考"对这些模型是多余的——它们在任何 Prompt 下都会自动执行内部推理链。

**被吞噬的部分**：通用的 CoT 指令（"逐步思考"、"请推理"）在推理模型上不再需要。——这些指令是一种"元指令"（教模型如何思考），推理模型已经在训练中学会了思考。继续写这些指令不仅浪费 Token，还可能因为与模型内置推理模式冲突而产生次优结果。

**仍然需要的部分**：领域特定的推理框架——"先分析时间复杂度，再分析空间复杂度，最后给出优化建议"——仍然需要显式书写。推理模型学会了"通用推理"，但没有学会"特定领域的推理顺序"。这个分工类似于编程语言（通用推理）和框架（领域特定推理顺序）的关系。

**工程建议**：从 2027 年起，默认假设模型具备通用推理能力。只在需要**约束推理方向**或**确保特定检查顺序**时，才书写显式的推理型 Prompt。

### 20.1.3 原生工具调用如何简化 Harness

当前主流 LLM 已支持原生工具调用（OpenAI 的 `tools` 参数、Anthropic 的 `tool_use`）。这意味着工具 Schema 的解析和验证由模型提供商的基础设施完成，而非 Harness。

**被简化的部分**：①工具 Schema 不需要被注入到 Prompt 中（模型接口原生支持）；②工具调用的解析和验证不需要 Harness 实现（API 直接返回结构化的 `tool_calls` 对象）；③工具调用的错误信息由模型提供商标准化。这些简化让第 9 章最小 Harness 的代码量可以进一步减少 30-50 行。

**Harness 仍然必须的部分**：①**权限控制**——原生工具调用不提供权限模型，仍然需要 Harness 在工具执行前检查能力清单；②**沙箱**——工具在模型提供商的服务器上被"调用"，但在用户的服务器上被"执行"——执行环境的隔离仍然是 Harness 的职责；③**人机交互**——审批模式、计划模式、中断语义（第 11 章）与工具调用是正交的——无论工具是谁调用的，审批流程都需要 Harness 实现。

## 20.2 "Engineering"会消失吗

如果模型能力持续进化，今天需要精心设计的 Prompt、Context 压缩、Harness 权限、Loop 终止条件——未来是否都会被模型内置的能力替代？"工程"是否会消失？

### 20.2.1 抽象层级的迁移规律

回顾计算史：每种新技术都创造了一个新的抽象层，然后被下一层技术"吞噬"。

```
汇编语言 → 编译器   （编译器吞噬了汇编层的工程）
SQL 数据库 → ORM    （ORM 吞噬了 SQL 层的部分工程）
手动内存管理 → GC   （GC 吞噬了内存管理的工程）
Prompt 工程 → ?     （? 是否会吞噬 Prompt 工程？）
```

本书的四层洋葱图——Prompt → Context → Harness → Loop——很可能遵循相同的迁移规律：下层的工程问题被模型能力"吞掉"后，工程焦点向上层迁移。

**当前正在发生的迁移**：
- Token 层（Prompt Engineering）：被推理模型吞噬了通用 CoT 部分，但领域特定的 Prompt 设计仍然需要。
- Window 层（Context Engineering）：RAG 正在被长上下文挤压，但权限控制和密度管理仍然需要。
- Runtime 层（Harness Engineering）：原生工具调用简化了 Harness，但权限、沙箱、HCI 仍然需要。
- System 层（Loop Engineering）：当前最不可能被吞噬的层——自主任务规划和学习循环需要外部系统支持。

### 20.2.2 从"调模型"到"建系统"的不可逆趋势

四层工程的核心趋势是从"调模型"（让模型做正确的事情）转向"建系统"（设计模型在其中运行的上下文、工具和循环系统）。

**"调模型"时代的特征**（2020–2024）：工程主要集中在 Prompt 层——写更好的 Prompt、选合适的采样参数、设计 Few-shot 样例。模型的输出质量高度依赖 Prompt 技巧。

**"建系统"时代的特征**（2025–至今）：模型的基础能力（推理、工具使用、长上下文）已经足够强大。瓶颈不再是"模型能不能做"，而是"系统能不能正确地把任务分配给模型、模型输出的结果能不能被正确执行"。工程重心从 Prompt 层上移到 Context 层和 Harness 层——如何管理上下文窗口中的信息密度（Context Engineering）、如何设计安全的工具执行环境（Harness Engineering）、如何编排长时间运行的自主任务（Loop Engineering）。

这个迁移是不可逆的——模型能力只会继续提升，不会倒退。这意味着提示词工程师的角色正在演变为"Agent 系统工程师"——需要理解四层工程的全栈知识。

### 20.2.3 工程师角色的演化

基于前文的迁移趋势，我们可以预见五年后的 Agent 工程师角色：

**今天的角色**：提示词工程师 → 主要工作在 Prompt 层，写提示词、调参数。输出是 Prompt 文本。

**未来演变路线**：提示词工程师 → Context 工程师 → Harness 工程师 → Agent 架构师。每一步的演进都是抽象层级的上升——从操控 Token 到管理上下文窗口到设计运行时环境到编排自主系统。

**Agent 架构师的核心能力**（五年后）：①**系统设计**——能够将用户需求分解为四层工程的协同设计；②**安全与治理**——理解 Prompt Injection 防御体系、权限模型和数据隐私；③**评估与迭代**——构建 Golden Set、设计评估流水线、持续优化系统性能。

本书的 20 章正是为了培养这样的全栈 Agent 架构师而设计的——从最底层的 Token 物理（第 2 章）到最上层的系统展望（第 20 章）。

```
全书回顾：四层洋葱图的完整旅程

第零层（导论）：为什么需要四种工程（第 1 章）
   ↓
第一层 Token  → Prompt Engineering（第 2–4 章）
   │              理解 Token 物理 → 掌握设计模式 → 工程化生产
   ↓
第二层 Window → Context Engineering（第 5–8 章）
   │              定义上下文五元组 → 构建 RAG → 记忆系统 → 工具上下文
   ↓
第三层 Runtime → Harness Engineering（第 9–12 章）
   │              设计 Harness 骨架 → 工具系统 → 人机交互 → 可观测性
   ↓
第四层 System  → Loop Engineering（第 13–16 章）
   │              循环拓扑 → 评估控制 → 长程恢复 → 学习型循环
   ↓
第五层（综合）：端到端案例 → 评估基准 → 安全伦理 → 展望（第 17–20 章）
```

## 开放问题

1. **下一个被吞噬的层。** 哪个层最有可能在两年内被模型能力吞噬？是 Prompt 层（推理模型吞噬 CoT）还是 Context 层（长上下文吞噬 RAG）？还是都不被吞噬，而是所有层的工程价值趋于均等？

2. **工程消失 vs 工程迁移。** "Engineering will disappear"是一个伪命题——工程不会消失，只会从低层迁移到更高层。Prompt 工程消失后，Harness 工程的复杂度会增加（因为需要管理更多样的工具）。这个迁移会持续到哪一层才是终点？

3. **全栈 Agent 架构师的培养。** 本书试图培养一个全栈智能体系统架构师。这需要同时理解 NLP（Prompt）、IR（Context）、系统设计（Harness）和分布式计算（Loop）四个领域的知识。这样的人才培养周期多长？是否有更高效的培养路径？
