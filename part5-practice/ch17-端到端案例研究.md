# 第 17 章 端到端案例研究

> **问题陈述**：前 16 章将智能体系统拆解为 Prompt、Context、Harness、Loop 四个工程层次分别讲解。然而，实际构建 Agent 时这四个层次不是独立设计的——它们在同一个系统中协同工作。本章通过三个真实案例（企业知识助手、编码 Agent、科研自动化），展示如何综合运用四层工程来构建一个完整的 Agent 系统。

---

## 17.1 案例 A：企业知识助手（重 Context）

### 17.1.1 需求与边界

**场景**：一家中大型企业需要一个内部知识助手，帮助员工查询公司政策、技术文档、项目历史和团队信息。

**需求**：①支持自然语言问诊（"报销流程是什么？"、"Team X 的 API 接口文档在哪里？"）；②答案必须引用可信来源（[来源1] 标记）；③权限控制——不同部门的员工看到的信息范围不同。

**四层工程协同**：
- **Prompt 层（第 2–4 章）**：系统提示词 $P$ 定义角色为"企业知识助手"和输出格式（JSON + 引用标记）。
- **Context 层（第 5–8 章）**： $R$ 分量承载向量检索结果， $S$ 分量记录用户角色和部门信息。
- **Harness 层（第 9–12 章）**：工具系统提供 `search_docs`、`query_knowledge_base` 等原子工具。
- **Loop 层（第 13–16 章）**：ReAct 循环——搜索→阅读→回答。

**边界条件**：①不处理涉及第三方机密信息的查询；②不执行代码或修改文档；③单次响应时间 ≤ 5 秒。

### 17.1.2 索引与权限的耦合设计

企业知识助手最核心的设计挑战是：**索引（谁看什么）和权限（谁能看什么）是耦合的**。同一个搜索查询，研发部门员工应看到技术设计文档，市场部门员工应看到产品手册。

解决方案：在索引时，为每个文档块附加访问权限标签（如 `department: engineering`、`level: public`）。检索时，用户的 $S$ 分量（user_id, department, role）被传入检索函数，检索结果根据用户权限过滤后仅返回被授权的文档块。

```yaml
# 索引文档的权限标签示例
documents:
  - id: "doc_001"
    title: "内部API设计规范"
    content: "..."
    permissions:
      departments: ["engineering", "product"]
      levels: ["confidential"]
  - id: "doc_002"
    title: "员工报销流程"
    content: "..."
    permissions:
      departments: ["all"]
      levels: ["public"]
```

检索时，权限过滤发生在 RRF 融合之前：向量检索和 BM25 召回的结果先经过权限过滤器，再进入 RRF 融合。这样即使一个文档与查询高度相关，如果用户无权访问，它也不会出现在最终的 $R$ 分量中。

### 17.1.3 评估指标与上线灰度

**评估指标**：①**检索召回率@5**（> 85%）；②**引用准确率**（> 95%——模型输出的引用必须在 $R$ 中有对应原文）；③**权限拒绝率**（< 0.1%——非授权文档被模型输出的比例）。

**上线灰度策略**：①**Shadow Mode**（影子模式）——新系统与旧系统并行运行，比较输出质量（2 周）；②**Canary Deployment**（金丝雀部署）——5% 的员工先使用新系统，监控引用幻觉率和用户满意度（1 周）；③**全量上线**——逐步扩展到 100% 用户。

> **失败案例**：某团队在影子模式阶段发现新系统的引用准确率远低于预期（78% vs 目标 95%）。排查后发现根因在 Loop 层——ReAct 循环中，模型在搜索后生成的回答没有正确使用引用标记。修复方案不是在 Prompt 层加更多指令（已加过 3 轮），而是在 Harness 层添加一个"引用验证"步骤：模型生成回答后，Harness 自动检查回答中的每个 `[来源N]` 是否在 $R$ 中有对应原文。如果发现无对应原文的引用，Harness 拒绝该回答并要求模型重新生成。

## 17.2 案例 B：编码 Agent（重 Harness）

### 17.2.1 需求与边界

**场景**：一个面向开发者的编码 Agent，帮助编写、审查和重构代码。

**需求**：①理解项目结构和代码逻辑；②执行代码片段以验证正确性；③修改多个文件时保持一致性；④支持 Git 操作和 PR 提交。

**四层工程协同**：
- **Prompt 层**：RTF 三段式——角色是"资深 Python 工程师"，任务为"审查代码变更"，格式为"问题列表"。
- **Context 层**： $O$ 分量承载代码执行结果（编译器输出、测试结果）， $R$ 分量承载相关代码搜索。
- **Harness 层**：工具系统提供 `read_file`、`write_file`、`run_tests`、`git_commit` 等工具；沙箱使用 Docker 容器隔离代码执行。
- **Loop 层**：Plan-Execute 范式——Agent 先输出执行计划（修改哪些文件、修改什么内容），用户批准后执行。

### 17.2.2 工具集裁剪

编码 Agent 的工具集需要精心裁剪。过多工具会增加 Token 消耗和模型选择开销，过少工具则限制 Agent 的能力。

裁剪后的核心工具集（10 个）：
- 文件操作（4 个）：`read_file`、`write_file`、`list_dir`、`search_in_files`
- 代码执行（2 个）：`run_command`、`run_tests`
- Git 操作（3 个）：`git_diff`、`git_commit`、`git_status`
- 外部查询（1 个）：`search_web_docs`

不包含的工具：`delete_file`（风险太高，由人工操作）、`install_package`（环境预配置，不需要 Agent 动态安装）、`deploy`（CI/CD 管线的职责，Agent 不触碰）。

### 17.2.3 沙箱与代码执行

代码执行是编码 Agent 最大的安全风险。沙箱设计必须同时满足**安全性**（恶意代码不影响宿主）和**功能性**（正常代码需要访问文件系统和网络）。

方案：每个用户会话创建一个独立的 Docker 容器，容器内预装项目依赖。代码执行时，Agent 将代码写入容器内的临时目录，Harness 使用 `docker exec` 执行代码。执行结果（stdout + stderr + exit_code）写回 $O$ 分量。容器的生命周期与会话绑定——会话结束后容器自动销毁。

容器配置：①**资源限制**——CPU 1 核、内存 512MB、磁盘 1GB；②**网络限制**——仅允许访问白名单域名（如 pypi.org、github.com）；③**超时控制**——单次代码执行超过 60 秒自动终止。

### 17.2.4 与 IDE 的集成

编码 Agent 的输出需要与 IDE（VS Code、JetBrains）集成。三种集成模式：①**Diff 展示**——Agent 修改文件后，IDE 显示修改的 diff（绿色新增/红色删除），用户逐块确认或拒绝；②**内联建议**——Agent 在光标位置提供代码建议，用户 Tab 接受；③**终端集成**——Agent 在 IDE 内嵌终端中执行命令，用户实时查看输出。

集成方式：Agent 通过 Harness 的文件系统工具修改文件，IDE 通过文件系统的 File Watcher 检测变更并刷新 diff 展示——不需要修改 IDE 本身的集成接口。这是"文件系统作为共享记忆"（第 10 章）的典型应用。

> **失败案例**：某编码 Agent 在沙箱中执行 `git push` 失败，因为容器内没有配置 SSH 密钥。Agent 多次重试失败后（每次消耗约 2,000 Token），将错误信息写回 $O$ ，下一轮 LLM 调用被错误信息误导，生成了一个"手动修复建议"而非继续推进。根因是工具 `git_commit` 的权限配置不完整（只配了本地权限，未配远程 push 权限）。修复方案：在 Harness 层的权限模型中，为 `git_commit` 补充远程操作权限声明，同时修改 $O$ 的错误信息写回策略（第 8 章的静默策略——临时性错误由 Harness 处理，不写回 $O$ 污染上下文）。

## 17.3 案例 C：科研自动化（重 Loop）

### 17.3.1 需求与边界

**场景**：一个科研自动化 Agent，帮助研究人员完成文献检索→实验设计→代码实现→报告生成的全流程。

**需求**：①从用户的研究方向描述出发，检索相关文献并生成综述；②根据文献中的方法，设计实验方案并生成实验代码；③执行实验、分析结果、生成报告。

**四层工程协同**：
- **Prompt 层**：Few-shot CoT——Agent 需要多步推理（"基于这些论文的方法，设计一个消融实验"）。
- **Context 层**： $R$ 分量持续注入最新检索的论文摘要； $H$ 分量承担整个研究过程的对话历史（可能持续数天）。
- **Harness 层**：工具系统提供 `search_papers`、`read_pdf`、`write_code`、`run_experiment`、`generate_report`。
- **Loop 层**：树搜索（Tree Search）——Agent 在文献检索阶段需要探索多个研究方向。

### 17.3.2 长程记忆与可复现性

科研自动化任务可能持续数天。Agent 需要长程记忆来记住用户的研究偏好和已完成的实验。

**长程记忆设计**（基于第 7 章的框架）：
- 事件流记录：每次实验运行的时间、参数和结果（Append-only）。
- 结构化记忆：用户偏好（"优先使用 PyTorch"、"图表用 Seaborn 风格"）、已阅读的论文列表。
- 增量摘要：每 10 步对话后生成摘要，压缩 $H$ 分量。

**可复现性设计**：科研自动化的核心要求是实验结果可复现。Harness 应为每次实验运行保存完整的"可复现包"——包含实验代码、依赖版本、输入数据和运行日志。当 Agent 生成报告中的"实验结果表明..."时，报告应附带可复现包的标识符（如 Git commit hash）。

### 17.3.3 失败案例剖析

**失败案例：思维链断裂导致实验设计偏离**

一个科研自动化 Agent 在第 5 步生成了一个实验设计——"对比模型 A 和模型 B 在数据集 X 上的准确率"。第 6 步开始编码实现，但 Agent 在第 8 步时遇到了一个技术问题（某个库版本不兼容），解决这个问题消耗了 3 步。到第 11 步时，Agent 已经完全忘记了原始的实验设计目标——它开始"顺手"优化代码性能，而非完成实验对比。

**根因分析**（跨层联动）：
- **Loop 层**：Reflexion 在技术问题处理步骤中没有反思"我是否还在推进原始目标"。
- **Prompt 层**：系统提示词 $P$ 中的任务描述被工具调用的 Observation 逐渐"挤出"上下文窗口。
- **Context 层**： $H$ 分量中的原始任务描述在第 8 步的技术问题讨论中被稀释。

**修复方案**（四层联动）：
| 层 | 修复措施 |
|----|---------|
| Prompt | 在 $P$ 中增加"实验设计的核心目标"段落，在截断时被保护（不参与滚动窗口） |
| Context | 每步将当前的子目标写入 $S$ 分量（不参与滚动），每 3 步与 $P$ 中的原始目标做一致性检查 |
| Harness | 在调度循环中增加"目标锚定"步骤——每次工具调用前输出当前子目标摘要 |
| Loop | Plan-Execute 范式替代 ReAct——先输出完整的实验设计计划，每步对照计划执行 |

修复后，Agent 在面对技术问题时不会偏离原始目标——Plan-Execute 范式的显式计划让 Agent 始终保持"我应该在哪个子目标的上下文中处理这个问题"的意识。

## 开放问题

1. **案例的分层可迁移性。** 企业知识助手（重 Context）的设计模式是否可以迁移到编码 Agent 场景？答案是肯定的——编码 Agent 也可以使用权限过滤来限制对工程目录的访问。

2. **端到端评估 vs 分层评估。** 第 18 章将讨论四层评估金字塔。端到端评估（"任务完成了没有"）回答"好不好用"；分层评估（"哪一层出了问题"）回答"为什么不好用"。实践中两者需要同时存在。

3. **案例的行业泛化。** 三个案例覆盖了企业内部、编码、科研三个场景。如果读者需要在医疗、法律、金融领域构建 Agent，这三个案例的设计模式是否可以复用？

## 练习

### 思考题

1. 从三个案例中选择一个你最熟悉的场景，使用第 1 章的四层洋葱图框架分析它的四层工程分配——哪一层的工作量最大？哪一层最容易被忽视？

2. 企业知识助手案例中，权限过滤发生在 RRF 融合之前。如果将权限过滤放到 RRF 融合之后（即先检索再过滤），会有什么优缺点？对用户体验有何影响？

3. 科研自动化案例中，Plan-Execute 范式替代 ReAct 后减少了目标漂移。但这种替代也增加了计划阶段的 Token 消耗。对于"文献检索"这个子任务（探索性、不可预测性高），Plan-Execute 是否仍然适用？还是不适用？为什么？

### 动手题

1. 为企业知识助手实现权限过滤函数：输入用户信息和文档列表，输出用户有权访问的文档子集。验收标准：给定 5 篇文档（权限标签各异）和 3 个不同部门的用户，每个用户只能看到对应的文档。

2. 为编码 Agent 实现 Docker 沙箱中的代码执行函数：将代码写入临时文件，在容器中执行，返回 stdout/stderr/exit_code。验收标准：一个打印 "hello" 的 Python 脚本返回 "hello"，一个语法错误的脚本返回非零 exit_code。

3. 为科研自动化 Agent 实现"目标锚定"步骤：在每次工具调用前，输出当前正在推进的子目标摘要。验收标准：一个 5 步任务的输出中，每一步都包含"当前目标：[子目标的名称]"
