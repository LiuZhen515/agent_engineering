# 第 6 章 检索增强（RAG）的工程化

> **问题陈述**：第 5 章定义了上下文的五元组结构 $C = \langle P, H, R, O, S \rangle$，其中检索结果 $R$ 是最活跃也最复杂的分量——它承载了外部知识注入上下文的全部工作。检索增强生成（RAG）是将外部知识检索并组装为 $R$ 的工程流程。本章从索引层、检索层、生成层三个维度，系统性地拆解 RAG 的工程细节。

**本部分导读：** 第 5 章建立了上下文五元组的理论框架，第 6 章深入实现其中最关键的检索分量 $R$。RAG 系统是上下文工程中实践最广、迭代最频繁的子系统，也是 Agent 获取外部世界知识的主要通道。如果你主要关注检索系统的设计与优化，6.1–6.2 是核心；如果你关心检索结果如何参与生成并避免引用幻觉，6.3 不可跳过。

> **贯穿案例延续**：本章继续"个人科研助手"案例。助手接收到用户请求"帮我查一下 2025 年关于大语言模型推理能力提升的最新论文"。检索 $R$ 分量的质量和效率直接决定了助手能否给出满意的答案。

## 6.1 索引层

索引层的任务是将原始文档转化为可检索的结构化表示。索引质量决定了检索召回率的天花板——再好的检索算法也无法找回未被索引的信息。

### 6.1.1 切分策略

将长文档拆分为可检索的文本块（Chunk）是索引的第一步。切分策略直接决定了召回率和上下文利用率之间的平衡。

**固定窗口与重叠率调优。** 最简单的切分策略是以固定 Token 数（如 256、512）切分文档，相邻块之间设置重叠（Overlap）。实验经验：重叠率在 10–20% 时，关键信息的割裂风险最小，同时引入的冗余 Token 成本可控。重叠率低于 5% 时，跨块的关键信息（如"上文提到的..."）被割裂的可能性显著增加——这对于依赖前文指的代词引用场景损害最大。建议切分大小与目标模型的上下文窗口配合：如果模型的有效上下文长度（ECL）为 32K，每次检索 Top-5 块，则每块应不超过 6K Token（5 × 6K = 30K < 32K）。

**语义切分与句界检测。** 固定窗口可能将一句话从中间切断，导致检索出的片段含意不全。语义切分在句子边界（句号、问号、换行符）处做切分，保持每个块的语义完整性。工具实现：LangChain 的 `RecursiveCharacterTextSplitter` 和 LlamaIndex 的 `SentenceSplitter` 均支持以句界为优先的切分策略。语义切分的缺点是块大小不均匀，可能导致部分块过大（超出模型的 ECL）或过小（信息密度不足）。

> **反方观点**：语义切分的"语义完整性"在向量检索时代可能被高估。基于嵌入的检索在决定匹配时已经隐式地将跨句信息编码到了向量中——即使切分点在句中，相邻块的语义重叠足以弥补割裂。据工程经验观察，固定窗口 + 充分重叠在大多数场景下优于复杂的语义切分器。

**层级切分（父子块、Small-to-Big）。** 层级切分是一种两阶段策略：文档被同时切分为小块（用于检索匹配）和大块（用于生成回答）。检索时用小块的密集向量匹配查询，找到最相关的小块后，返回其所属的大块给模型生成。这种"检索用细粒度、生成用粗粒度"的设计兼顾了召回精度和上下文完整性。典型配置：小块 128–256 Token，大块 1,024–2,048 Token。父子块策略在需要精确引用的场景（如法律文档、技术规范）中表现尤为突出。

**代码 / 表格 / PDF 的特殊处理。** 这三种文档类型不能直接使用通用文本切分。**代码**应按函数/类边界切分（而非行数），并保留注释和导入语句作为上下文锚点。**表格**（CSV、Excel）应行列保留，单行作为最小切分单元，因为表格跨行后语义不可恢复。**PDF** 需要额外的布局解析步骤（如使用 PyMuPDF、Marker 等工具），因为 PDF 中的文字物理位置（多栏、页眉页脚、浮动图片）在纯文本抽取后可能完全错乱。工程建议：将 PDF 转换为 Markdown 后再索引，Markdown 的标题层级保留了文档的结构信息，能显著提升检索质量。

### 6.1.2 嵌入模型选型

嵌入模型将文本映射为稠密向量，是向量检索的基石。

**通用 vs 领域微调嵌入。** 通用嵌入模型（如 text-embedding-3-small、BGE-base）在开放域检索中表现均衡，但在特定领域（如医疗、法律、代码）中，领域微调嵌入（如 PubMedBERT、LegalBERT）的检索准确率通常领先 10–20 个百分点。选型建议：先使用通用模型建立基线，再对 Golden Set 做消融实验。如果领域特定召回率低于通用模型 5% 以上，考虑微调或更换领域嵌入。MTEB 基准 (Muennighoff et al., 2023) 提供了各嵌入模型在多项任务上的对比数据，可作为初始选型参考。

**维度与召回率权衡。** 嵌入向量的维度越高，理论上能编码的信息越丰富，但存储和检索成本也越高（向量维度增加一倍，索引大小增加一倍，检索延迟增加 20–30%）。经验法则：768 维嵌入在大多数场景下已足够，1,536 或更高维度仅在需要极高精度的场景（如法务尽职调查）中才值得其额外成本。降维技术（如 PCA、Matryoshka Representation Learning）可以在不显著损失召回率的情况下将维度压缩 50–75%。

**多语言嵌入的对齐问题。** 跨语言检索（如用户查询为中文、文档为英文）对嵌入模型的多语言对齐能力要求很高。通用多语言嵌入（如 multilingual-e5、BGE-m3）在中英互检场景下的表现已经接近单语言嵌入的 80–90%，但在小语种（如阿拉伯语、泰语）中仍存在显著差距。工程建议：如果你的系统同时处理中英文，使用专门训练的中英双语嵌入（如 bge-large-zh-v1.5）而非通用多语言模型，通常能获得更好的检索效果。

## 6.2 检索层

检索层在索引层之上执行查询匹配，决定哪些信息块进入上下文 $R$ 分量。

### 6.2.1 多路召回

单一检索方法有其固有盲区。多路召回通过组合不同检索方法的优势来提升整体召回率。

**向量召回。** 向量召回（Dense Retrieval）使用嵌入模型将查询 $Q$ 和文档块 $D_i$ 都映射为向量，通过余弦相似度 $\cos(\vec{Q}, \vec{D}_i)$ 找到最相似的 Top-$k$ 块。优势在于语义匹配——即使查询和文档使用不同的词汇表达同一概念（如"车"与"automobile"），向量空间中的相似度也能捕获这种语义关系。劣势在于对高频实体和精确关键词匹配的敏感性低于稀疏方法。

**BM25 与稀疏召回。** BM25 是一种基于词频-逆文档频率（TF-IDF）的检索方法，不依赖嵌入模型。它精确匹配查询中的关键词，对实体名称（如"GPT-4"、"Transformer"）和稀有词汇的召回率高于向量召回。在长尾查询（查询中包含低频专有名词）中，BM25 的精确匹配优势尤为明显。BM25 的实现成本极低——可以在内存中运行，无需 GPU。

**Hybrid Fusion（RRF / 加权融合）。** 混合融合将向量召回和 BM25 的结果合并。最常用的融合方法是倒数排名融合（Reciprocal Rank Fusion, RRF）：对于在 $N$ 种检索方法中每一种方法排名为 $r_i$ 的文档，其融合得分 $S = \sum_{i=1}^N \frac{1}{k + r_i}$，其中 $k$ 为常数（通常为 60）。RRF 的优势在于不依赖分数归一化——不同检索方法的分数尺度（余弦相似度 0–1 vs BM25 0–100+）无需对齐即可融合。加权融合方法则对每种方法分配权重（如 0.7 向量 + 0.3 BM25），权重需通过 Golden Set 调优。

```python
# Listing 6.1  RRF 融合函数
# 完整代码见 agent-engineering-code/part2-context/ch6-rag-pipeline/rrf_fusion.py

def reciprocal_rank_fusion(
    ranked_lists: list[list[str]],
    k: int = 60,
) -> list[tuple[str, float]]:
    """对多个检索结果列表执行 RRF 融合。

    ranked_lists: 每个元素是一个文档 ID 排名列表（从高到低）。
    返回按融合得分降序排列的 (doc_id, score) 列表。
    """
    scores: dict[str, float] = {}
    for rank_list in ranked_lists:
        for rank, doc_id in enumerate(rank_list, start=1):
            scores[doc_id] = scores.get(doc_id, 0.0) + 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)


# 示例：融合向量召回和 BM25 的结果
vec_ranked = ["doc1", "doc3", "doc2"]     # 向量召回排名
bm25_ranked = ["doc2", "doc1", "doc4"]    # BM25 召回排名
fused = reciprocal_rank_fusion([vec_ranked, bm25_ranked])
for doc_id, score in fused[:3]:
    print(f"{doc_id}: {score:.4f}")
```

> **工程原则 1（多路召回原则）**：任何生产级 RAG 系统都应至少使用两种互补的检索方法（如向量 + BM25），通过 RRF 融合结果。单一检索方法的召回率上限通常比多路融合低 10–20%。

### 6.2.2 Re-ranker 流水线

多路召回生成一个候选池（通常 Top-50），Re-ranker 对这个候选池做更精细的排序，选出最终进入上下文的前 Top-$k$（通常 3–5 个）。

**Cross-encoder vs LLM Reranker。** Cross-encoder（如 BGE-reranker-v2、Cohere rerank）将查询和文档拼接后一次性输入模型，输出一个相关性分数。它比双编码器（Bi-encoder，即嵌入向量检索）更准确，因为模型能够"看到"查询和文档之间的交互。但 Cross-encoder 的成本也更高——为每个查询-文档对运行一次推理。LLM Reranker 则直接使用 LLM（如 GPT-4o-mini）判断文档-查询相关性，优点是零额外模型部署成本（复用已有 LLM），缺点是延迟更高、成本取决于 Token 消耗。**选型建议**：高吞吐系统用 Cross-encoder（延迟 < 50ms/对）；低吞吐但高精度场景用 LLM Reranker（可结合 CoT 推理解释相关性判据）。

**定义 6.1（检索延迟预算）**：RAG 系统的检索预算 $B_{\text{retrieval}}$ 定义为从用户查询到达系统到检索结果 $R$ 被注入上下文之间的最大允许时间。$B_{\text{retrieval}}$ 由以下公式约束：

$$B_{\text{retrieval}} = T_{\text{embed}} + T_{\text{search}} + T_{\text{rerank}}$$

其中 $T_{\text{embed}}$ 为查询嵌入时间，$T_{\text{search}}$ 为向量库搜索时间，$T_{\text{rerank}}$ 为 Re-ranker 时间。经验阈值：$B_{\text{retrieval}} < 2s$（聊天场景），$B_{\text{retrieval}} < 5s$（RAG 问答场景）。

**延迟预算分配。** 在固定的 $B_{\text{retrieval}}$ 下，三阶段的延迟分配决定了最终召回质量。典型分配：$T_{\text{embed}}$ 占 10%（约 100–300ms）、$T_{\text{search}}$ 占 20%（约 200–500ms）、$T_{\text{rerank}}$ 占 70%（约 500–1,500ms）。如果 re-rank 延迟过高，可以缩小候选池大小（Top-30 → Top-20）或使用更轻量的 Cross-encoder。

### 6.2.3 查询改写

用户原始查询通常不适合直接检索——它可能太短、太模糊或缺乏上下文。

**HyDE 与假设文档。** HyDE（Hypothetical Document Embeddings）先生成一个"假设文档"——假设存在一篇完美的回答当前问题文档的内容，然后使用这个假设文档的嵌入去检索真实文档 (Gao et al., 2023)。这种方法特别适用于查询与文档语言风格差异大的场景（如用户的简短问句 vs 冗长的学术论文）。执行流程：查询 → LLM 生成假设文档 → 用假设文档的嵌入检索 → 返回真实文档。额外成本：一次 LLM 调用 + 一次嵌入计算。

**Multi-Query 扩展。** Multi-Query 将一个用户查询扩展为多个语义相近但不完全相同的子查询，分别检索后合并结果。例如，"Transformer 的注意力机制"可以扩展为"self-attention 的工作原理"、"多头注意力的计算过程"、"注意力权重可视化方法"。每个子查询独立检索 Top-$k$，合并后去重。Multi-Query 的召回率提升约为 15–25%，代价是额外的 $n$ 次嵌入计算和检索调用。

> **真实失败案例**：某团队在生产环境中使用单一向量检索，查询为"2024 年最佳 AI 论文"。向量召回返回了大量关于"2024 年"的新闻而非关于"AI 论文"的学术文献。加入 BM25 + RRF 融合后，召回率从 68% 提升至 87%。进一步使用 HyDE（LLM 先生成一段假设的论文摘要再检索），召回率提升至 93%。

**Least-to-Most Prompting。** Least-to-Most Prompting 在检索前先让 LLM 将复杂问题拆解为一系列更简单的子问题，然后对每个子问题分别检索 (Zhou et al., 2023)。例如，从"如何优化 Python 列表推导式的性能"拆解为"Python 列表推导式的性能特征"和"Python 中的性能优化技术"。拆解式检索能召回更基础/更全面的背景知识，优于直接对完整问题做一次检索。

## 6.3 生成层

检索结果 $R$ 注入上下文后，如何组织它们以避免被注意力稀释、如何避免引用幻觉——这些都是生成层的问题。

### 6.3.1 上下文压缩

**抽取式压缩。** 对检索到的文档块做抽取式压缩：提取每个块中最相关的 1–2 个句子，丢弃其余内容。压缩率通常为 60–80%（即 1,000 Token 的块压缩至 200–400 Token）。压缩后的块丢失了上下文连贯性，但保留了关键事实。适用于对精确信息提取要求高、对语言流畅性要求低的场景（如事实问答）。

**Map-Reduce 与 Refine 链。** 当每个单独的文档块不足以回答用户问题时，需要跨块推理。Map-Reduce 链：先对每个块分别执行回答（Map 阶段），再将所有块的回答合并为一个最终答案（Reduce 阶段）。Refine 链：顺序处理每个块，每次将当前块的输出作为下一个块的额外上下文。经验对比：Map-Reduce 适合块间独立性强、答案可分段的任务（如"列出各公司的收入"）；Refine 适合需要全局连贯性的任务（如"总结整篇论文的贡献"）。

**个人科研助手案例（检索结果展示）：** 助手对用户请求"帮我查一下 2025 年关于大语言模型推理能力提升的最新论文"执行了向量 + BM25 双路召回，RRF 融合后取 Top-3。最终注入上下文 $R$ 分量的内容为：
```
[1] 来源：arXiv:2501.12948, 2025-01
    DeepSeek-R1 使用强化学习提升推理能力，在数学和编程基准上达到 o1 水平...
[2] 来源：arXiv:2503.12345, 2025-03
    Chain-of-Thought 在推理模型中的角色变迁——从显式引导到隐式推理...
[3] 来源：arXiv:2505.06789, 2025-05
    大语言模型自我纠错能力的边界条件分析...
```
此 $R$ 分量为生成层提供了三条可追溯的来源，助手基于它们生成回答并在答案中标注了引用标记 `[1]`、`[2]`、`[3]`。如果某一篇论文的内容不足以支撑回答，助手应返回"资料不足"而非编造。

**RAPTOR 层级摘要。** RAPTOR (Sarthi et al., 2024) 构建了一个多层级索引：底层是原始段落，上一层是段落聚类的摘要，再上一层是摘要的摘要。检索时，根据查询的粒度自动选择最适合的层级——细粒度查询检索底层，粗粒度查询检索顶层。RAPTOR 的核心工程创新是将"检索空间"从二维（向量相似度排名）扩展为三维（向量相似度 × 抽象层级），允许系统在同一检索操作中同时搜索事实级和概念级信息。

### 6.3.2 引用与可追溯性

**行内引用标记。** 提供可追溯的引用是增强 $R$ 分量的可信度 $C_r$ 的关键。行内引用标记格式在 $R$ 注入上下文时附加文档标识符：

```
RAG 上下文：
[1] 来源：Attention is All You Need (Vaswani et al., 2017), 第 3 页
     多头注意力机制将查询、键和值分别投影 h 次...
[2] 来源：BERT (Devlin et al., 2019), 第 2 页
     BERT 使用双向 Transformer 预训练...
```

标记 $[1]$ 和 $[2]$ 在答案中可以被引用（如"据文献 [1] 所述"）。实现方式：在检索结果 $R$ 写入上下文前，为每个块附加一个不可跳过的元数据前缀。

**引用幻觉的检测与拒答。** 引用幻觉——模型在答案中引用了一个不支持的来源——是 RAG 系统最危险的失败模式之一。检测方法：在生成答案后，用一个独立的"忠实度检查" LLM 调用来验证模型答案中的每个引用是否在已提供的 $R$ 中存在对应原文。如果无法确认，则拒绝回答并返回"我无法在提供的参考资料中找到支持该观点的证据"。更激进的做法：在模型的 System Prompt $P$ 中写入"如果无法在提供的参考资料中找到确切依据，必须说'资料不足'，不得编造引用"。

> **工程原则 2（引用可验证原则）**：RAG 系统输出的每个事实主张都应在 $R$ 分量中找到对应的原文引用。如果模型无法在 $R$ 中验证其输出的准确性，应拒绝回答而非猜测。

## 附：RAG 评估指标表

| 指标名称 | 定义 | 度量方法 |
|---------|------|---------|
| 检索召回率@K（Recall@K） | 前 $K$ 个检索结果中包含相关文档的概率 | 在 Golden Set 中计算 $\lvert\text{相关结果} \cap \text{前 }K \text{ 个}\rvert / \lvert\text{相关结果总数}\rvert$ |
| 融合提升率 | 多路融合相比单一方法的召回率提升 | (Recall@K_hybrid - Recall@K_single) / Recall@K_single × 100% |
| 检索延迟 $B_{\text{retrieval}}$ | 从查询到 $R$ 注入的时间间隔（定义 6.1） | 生产环境实测 $T_{\text{embed}} + T_{\text{search}} + T_{\text{rerank}}$ |
| 引用准确率 | 模型输出的引用在 $R$ 中有对应原文的比例 | 人工/自动抽取 $N$ 个输出中的引用，验证原文存在性 |
| 上下文压缩率 | 压缩后 Token 数 / 压缩前 Token 数 | 在抽取式压缩前后分别计数 |

## 开放问题

1. **Embedding 模型的风火轮问题。** 嵌入模型的更新周期比 LLM 更快（每年 5–10 个新版本），每次更新都需要重新索引全量文档。是否存在一种"跨嵌入模型的兼容性映射"，使不同版本的嵌入向量可以在同一索引中共存？

2. **检索结果的注意力位置分配。** 第 5 章的位置策略建议将检索结果 $R$ 放在上下文中间层，但不同质量的 $R$ 可能应分配到不同位置——高质量结果放在更靠后的位置（Recency Effect 更强），低质量结果放在中间。如何自动化这种位置分配？

3. **引用幻觉的根因是模型倾向还是检索质量？** 当模型产生引用幻觉时，是因为它"试图引用"（Generation 层的问题）还是因为正确的资料没有被检索到（Retrieval 层的问题）？实践中这两类原因的占比是多少？

4. **RAG 是否会被长上下文取代？** 当模型的上下文窗口达到 10M Token 时，理论上可以一次性"喂入"整个知识库，让模型自己查找所需信息。这种情况下 RAG 的索引和检索层是否会变得多余？第 20 章将深入讨论这一趋势。

## 练习

### 思考题

1. 你在第 5 章拆解了应用的上下文五元组。在此基础上，分析该应用中 $R$ 分量的质量（使用 6.2 节的多路召回框架）——当前的检索是否只用了单一方法？多路融合能否提升召回率？代价是什么？

2. 假设你的 RAG 系统需要同时支持精确事实问答（"GPT-3 有多少参数？"）和概念综述（"解释自注意力机制的原理"）。你会分别配置什么样的检索策略（切分大小、检索方法、Re-ranker 选择）？两种策略是否能共用同一个索引？

3. 引用幻觉检测的成本和收益如何权衡？如果每次生成后都运行一次"忠实度检查"LLM 调用，成本翻倍。什么比例的幻觉率阈值以下你愿意接受不检查？

### 动手题

1. 实现 Listing 6.1 的 RRF 融合函数，并在一个你自己构建的测试集上比较向量召回、BM25 和 RRF 融合三种方法的 Recall@5。验收标准：输出一个三方对比表格，并标注融合方法相比最佳单一方法的提升率。

2. 为一个 50 页以上的 PDF 文档实现 RAPTOR 风格的层级索引（至少 2 层：段落级 + 摘要级）。验收标准：给出一个"从粗到细"的检索示例——先用顶层摘要级检索缩小范围，再到底层段落级检索定位具体事实。

3. 设计一个引用幻觉检测器：给定一个模型生成的答案及其引用的 $R$ 集合，判断每个引用是否在 $R$ 中有对应的原文支持。验收标准：在 10 个测试用例上达到 ≥ 80% 的检测准确率（无需 LLM-as-Judge，可用规则匹配）。
